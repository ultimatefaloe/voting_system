================================================================================
                            API ENDPOINTS (5)
================================================================================

1. POST /elections/{id}/votes
   Description: Submit single vote
   Auth: X-Voter-Token header
   Body: { position_id, candidate_id }
   Response: 201 or error

2. POST /elections/{id}/votes/batch
   Description: Batch voting (atomic)
   Auth: X-Voter-Token header
   Body: { votes: [...] }
   Response: 201 or error (all or none)

3. GET /elections/{id}/ballot
   Description: Get voter's ballot
   Auth: X-Voter-Token header
   Response: 200 with positions and selected candidates

4. GET /elections/{id}/results
   Description: Get published results
   Auth: Public/membership based
   Response: 200 with vote counts and percentages

5. GET /organizations/{id}/elections/{id}/voting-stats
   Description: Get voting statistics
   Auth: Bearer token (org membership)
   Response: 200 with turnout data

================================================================================
                          SECURITY ARCHITECTURE
================================================================================

ATOMIC TRANSACTIONS:
- DB::transaction() with 5 retry attempts
- All votes or none (ACID guarantees)
- Automatic rollback on error
- Transaction isolation level: Serializable

ROW-LEVEL LOCKING:
- lockForUpdate() for vote session
- Prevents concurrent duplicate sessions
- Handles race conditions at database level

DOUBLE-VOTE PREVENTION (3-Layer):
1. Structural: Unique constraint (election_id, voter_token)
2. Application: Query validates before save
3. Business Logic: Max votes per position enforced

TOKEN VALIDATION:
- Not expired
- Belongs to election
- One-time use
- Status check
